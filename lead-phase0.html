Executive Summary — MVS Scope

This environment houses the Minimum Viable System (MVS) logic for LEAD's Phase 0 prototype, structured for Business Finland grant evaluation. It demonstrates the core interaction loops between consumers, independent chefs, courier logistics, and the platform orchestration layer. All modules below represent live prototype logic — not mockups — and are designed to validate the four-way handshake model, result-oriented discovery UX, and geo-radius logistics routing as fundable technical differentiators.

The Four-Way Handshake — Logistics State Machine

PROTOTYPE LOGIC: The handshake FSM (Finite State Machine) represents the core orchestration model. A Webhook trigger fires on Chef acceptance, routing a geo_radius_ping to available couriers within the configured radius. The platform acts as the coordination authority at State 3.

State Transitions
STATE_1 Consumer → Indent Placed

Consumer submits an Indent referencing a Chef listing. Writes to active_indents_collection with order_state: "PENDING".

STATE_2 Chef → Indent Accepted

Chef acknowledges availability and confirms preparation timeline. State transitions to "CONFIRMED". Triggers Platform webhook listener.

STATE_3 Platform → Logistics Webhook

Platform dispatches webhook to logistics_webhook_routing. Computes geo_radius_ping from Chef coordinates and assigns courier_bounty_price.

STATE_4 Courier → Geo-Radius Assigned

Nearest available Courier within radius receives ping and accepts dispatch. Order state transitions to "IN_TRANSIT".

Core Schema

Backend Architecture — Database Collections

users_collection MongoDB
// Core identity & KYC layer
{
user_id: ObjectId,
// UUID v4, immutable
role_type: Enum[
"CONSUMER",
"CHEF",
"COURIER",
"ADMIN"
],
kyc_status: Enum[
"UNVERIFIED",
"PENDING",
"VERIFIED"
],
live_geo_location: {
type: "Point",
coordinates: [lng, lat]
} // GeoJSON
}
// Indexes
idx_1: { live_geo_location: "2dsphere" }
idx_2: { role_type: 1, kyc_status: 1 }
active_indents_collection MongoDB
// Live order state tracking
{
indent_id: ObjectId,
consumer_id: Ref(users),
chef_id: Ref(users),
courier_id: Ref(users) | null,
order_state: Enum[
"PENDING",
"CONFIRMED",
"DISPATCHED",
"IN_TRANSIT",
"DELIVERED",
"CANCELLED"
],
created_at: ISODate,
ttl_expires: ISODate
}
// Indexes
idx_1: { order_state: 1 }
idx_2: { consumer_id: 1, created_at: -1 }
ttl: { ttl_expires: 1 } // auto-expire
logistics_webhook_routing MongoDB
// Dispatch & geo-routing events
{
dispatch_id: ObjectId,
associated_indent_id:
Ref(active_indents),
geo_radius_ping: {
origin: GeoJSON,
radius_km: Float,
candidates_found: Int,
ping_timestamp: ISODate
},
courier_bounty_price: {
base_rate: Decimal128,
surge_multiplier: Float,
currency: "EUR"
},
webhook_status: Enum[
"FIRED",
"ACCEPTED",
"TIMEOUT"
]
}
// Indexes
idx_1: { associated_indent_id: 1 }
idx_2: { webhook_status: 1, ping_timestamp: -1 }
Entity Relationship Overview
users_collection (role: CONSUMER) ──[places]──→ active_indents_collection ←──[assigned_to]── users_collection (role: CHEF)
active_indents_collection ──[triggers]──→ logistics_webhook_routing ──[pings]──→ users_collection (role: COURIER, kyc: VERIFIED)
